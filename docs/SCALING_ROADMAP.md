# Goofish Monitor 扩容演进方案

本文档描述当前执行架构、已经完成的止血优化，以及从当前规模演进到 `1000` 任务级别的推荐路线。

## 当前约束

闲鱼监控的约束不是纯 CPU 或数据库问题，而是以下几类混合瓶颈：

- 搜索结果天然有噪音，必须使用精确型号关键词，不能退化为“大分类宽词搜索”
- 登录态、风控验证、可见浏览器接管都依赖浏览器上下文
- 任务如果同时触发，会迅速拉起大量 Playwright / 子进程，直接拖垮服务
- 单个任务执行链较长，包含搜索、详情页、AI 分析、图片下载、结果入库、通知发送

因此扩容路线必须坚持两点：

1. 保持“具体型号搜具体词”的精度
2. 将“任务数量”与“实际运行的爬虫执行单元数量”彻底解耦

## 已完成的阶段

### 阶段 0：止血

已完成：

- 相同 Cron 任务自动错峰
- 调度器全局并发上限
- 定时触发不再直接拉起一堆爬虫

问题：

- 仍然依赖 `spider_v2.py` 子进程
- 任务执行状态只有“运行中/未运行”，不够细

### 阶段 1：统一执行队列

已完成：

- 引入 `ExecutionQueueService`
- 调度器只负责 enqueue
- 手动启动和定时触发统一走队列
- 固定 worker 池消费任务
- 后台状态接口返回 `execution_queue`
- 任务接口返回 `is_queued / execution_state`

当前价值：

- 任务不会在同一时刻爆发式启动
- 队列、执行、调度至少进入统一模型

## 阶段 2：浏览器上下文复用

目标：

- 将“每个任务起一个 `spider_v2.py`”推进为“固定 worker 复用浏览器上下文”

建议实现：

- 新建 `ScrapeWorkerService`
- 每个 worker 常驻：
  - 1 个 Playwright browser
  - 1 个 browser context
  - 1 个默认登录态
- 队列中的任务由 worker 直接调用库内抓取函数，而不是再起子进程

需要改造：

- 将 `spider_v2.py` 中的“任务准备逻辑”抽出为可复用服务
- 将 `src.scraper.scrape_xianyu(task_config)` 接成库内直接调用
- 将日志输出从子进程 stdout 改为任务级 logger

收益：

- 降低子进程开销
- 降低浏览器反复启动成本
- 更适合几十到上百任务

## 阶段 3：队列持久化与失败调度

目标：

- 即使服务重启，也不会丢失待执行任务和冷却状态

建议实现：

- 持久化队列表：
  - `task_id`
  - `status`
  - `enqueued_at`
  - `started_at`
  - `finished_at`
  - `attempt`
  - `last_error`
  - `cooldown_until`
- 支持失败退避
- 支持风控冷却
- 支持登录失效挂起

收益：

- 任务不会在错误情况下反复自杀式重试
- 可以建立真正的“运行中心”

## 阶段 4：按任务组策略调度

目标：

- 同一分类/任务组内任务不再是简单平均调度

建议实现：

- 为 `category / group / task` 引入优先级
- 同组任务按预算带、活跃度、最近命中率动态调度
- 支持“热点任务优先”
- 支持“长时间没货的任务降频”

收益：

- 将有限执行资源优先投给最有价值的任务

## 阶段 5：1000 任务级别的目标形态

当任务量到 `1000` 左右时，推荐架构应为：

- 调度器：只做“哪些任务应入队”
- 持久化队列：负责排队、失败、冷却、优先级
- Worker Pool：少量固定 worker 常驻
- 浏览器上下文池：按账号/分类复用
- 结果处理流水线：
  - 搜索
  - 结果过滤
  - 详情获取
  - AI 分析
  - 入库
  - 通知
- 运行中心：统一展示
  - 队列长度
  - worker 状态
  - 风控任务
  - 冷却任务
  - 登录失效任务

## 明确不建议的路线

以下路线看似简单，但会在任务数量上来后快速失效：

- 每个任务定时直接起一个爬虫进程
- 用“扫地机器人”这种大词代替具体型号搜索
- 所有任务使用相同频率、相同优先级
- 所有错误立刻无限重试

## 当前建议优先级

如果继续开发，推荐按以下顺序推进：

1. 浏览器上下文复用
2. 队列持久化
3. 运行中心 UI
4. 任务组级调度策略

## 当前版本定位

当前版本已经从：

- 调度器直接拉起大量爬虫

演进为：

- 调度器入队
- 固定 worker 池执行

它适合：

- 十几个到几十个任务的稳定监控

它还不适合：

- 直接承载 `1000` 任务规模

后续扩容必须继续推进阶段 2 到阶段 4。
